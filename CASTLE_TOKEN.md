# X login failure — missing $castle_token (open)

Tracks the investigation into why logging into X/Twitter through the webview
proxy shows **"Please use X.com or official X apps to proceed with log in/
sign up"**, separate from the sidecar/TLS-fingerprint work in
`SETUP_SIDECAR.md`.

## Root cause: our own Castle.io stub strips the anti-fraud token X's server expects

Compared two HAR captures of the same `POST .../onboarding/web/actions/begin_login`
request: one from a real, working Firefox session directly against x.com, one
recorded at the sidecar's MITM layer (`/mitm/har`) going through our proxy.

- **Working request body:** `username_or_email=trailsairpark&$castle_token=<~15KB of encoded data>`
- **Broken request body:** `username_or_email=trailsairpark` — nothing else
- `Content-Length`: 15151 vs 31.

`$castle_token` is generated by Castle.io's behavioral/device-fingerprinting
SDK (`ondemand.castle.<hash>.js`), which X's backend apparently requires on
`begin_login` — a request missing it looks exactly like what a scripted,
non-browser client would send, and gets rejected.

We don't ship Castle's real code: `src/runtime/webview/proxy.ts`'s
`extractWebpackChunkStub()` intercepts that chunk and replaces every module
body with a no-op (keeping only the webpack registration wrapper, so the
bundle doesn't throw `ChunkLoadError` and cascade into breaking the rest of
the page). This was done because the real SDK was reported to crash "inside
the proxy iframe context (cross-origin parent access)" — but stubbing it
out means Castle's fingerprinting code never runs at all, so the token
X expects is just never there.

Also found in the same HAR diff, smaller but real: `Sec-Fetch-Site` was
`same-site` for a request where Origin and Host are both exactly `x.com` —
should be `same-origin`. `sidecar/mitm-proxy.mjs`'s `isSameSite()` only
distinguishes same-site/cross-site, no same-origin case. Not yet fixed.

## Attempted reproduction of the original crash: inconclusive, crash did not reproduce

Added `CASTLE_DEBUG_REAL=1` (`src/runtime/webview/proxy.ts`) to serve the
real, unstubbed Castle chunk instead of the no-op stub, for testing.

Test setup: extracted the real `ondemand.castle.c42626ea.js` from the
working reference HAR (same content hash as what our own proxy fetches, so
it's the same version), ran it through our actual proxy pipeline locally
(dev backend, `CASTLE_DEBUG_REAL=1`, a webview row created directly via DB
insert to skip needing a real login session), loaded the login page in a
genuine cross-origin iframe via Patchright (`--host-resolver-rules` mapping
the webview's slug hostname to `localhost:3000`, matching production's real
`{slug}.app.onetrueos.com` vs `app.app.onetrueos.com` origin split), typed a
username, clicked Continue.

**No crash across 7 runs.** Confirmed via `[castle] stub: none — returning
real script` in logs that the real script was genuinely being served each
time, not silently falling back to the stub. The page behaved normally
through username submission. Testing was cut short by X's own legitimate
rate-limiting ("We've temporarily limited your login") after repeated
automated attempts against the same test account — not a reproduction of
the original issue, just rate-limiting from the test methodology itself.

This contradicts the existing code comment's claim of a definite crash.
Possible explanations, none confirmed:

- The crash may only occur **later** in the flow (password entry, a 2FA/
  security-challenge step) — rate-limiting stopped testing before reaching
  that point.
- It may be specific to the **TLS sidecar's** real-Chrome/MITM pipeline
  rather than the plain local dev-backend path tested here (untested: this
  reproduction used direct undici fetches, not the sidecar).
- Castle's SDK is versioned by content hash in its filename
  (`ondemand.castle.c42626ea.js`) — if X shipped an update since whoever
  wrote the original stubbing comment observed the crash, this specific
  version may no longer have the bug.

## Next steps (not yet done)

  account), then push further into the flow with `CASTLE_DEBUG_REAL=1` —
  password entry, and ideally a real login attempt — watching for the
  crash at each step.
- If it still doesn't reproduce locally, try the same test through the
  **real deployed sidecar** (where the original crash was actually
  observed), since that's the most faithful environment to whatever
  originally motivated the stub.
- If the crash genuinely no longer reproduces, the real fix might just be
  **deleting the stub entirely** and letting Castle's real SDK run — worth
  confirming deliberately rather than assuming, given how confidently the
  original comment described the crash.
- Fix the `Sec-Fetch-Site: same-site` vs `same-origin` bug in
  `sidecar/mitm-proxy.mjs`'s `isSameSite()` regardless — smaller, but a
  real, confirmed bug independent of the castle_token question.
